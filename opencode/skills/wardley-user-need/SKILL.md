# Wardley Mapping — User Need

## Purpose

Extract a clear, concrete user need expressed as a real-world outcome.

## Core Rules

- ALWAYS identify the user
- ALWAYS identify the outcome
- REJECT vague/system-centric phrasing
- FORCE concrete wording
- DO NOT proceed without a valid user need

## Interaction

Ask:

- Who is the user?
- What are they trying to achieve?
- What does success look like?

## Constraints

Reject answers like:

- “manage X”
- “handle Y”
- “support Z”

Push toward:

- real-world outcomes
- observable success

## Challenge

If vague, respond with:

- “This is too abstract. What actually happens in the real world?”
- “What does the user get that they didn’t have before?”

## Exit Criteria

- One clear user need
- Expressed as an outcome

## Forbidden

- Moving to components
- Discussing architecture
- Discussing evolution
